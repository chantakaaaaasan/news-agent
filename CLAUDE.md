# CLAUDE.md — ニュースダイジェスト・エージェント運用規約

このリポジトリで作業するAI(Sonnet / Haiku 含む)への指示書。上位モデルの判断力を前提にせず、この規約に従えば同品質で構築・修正・運用できるように書いてある。**迷ったらこのファイルの規約を優先し、自分の判断で規約を逸脱しない。**

## 1. このシステムは何か

毎朝、RSS/Atomフィードからニュースを収集し、AIで要約してSlackに通知するパイプライン。

```
collect.py(収集) → main.py(既読除外) → summarize.py(要約) → notify.py(Slack通知) → state.json(既読保存)
```

## 2. 絶対に守る設計原則(逸脱禁止)

1. **依存ゼロ**: Python標準ライブラリのみ。`pip install` が必要なライブラリ(feedparser, requests等)を追加しない。追加したくなったら、まず標準ライブラリで書けないか3回考える。
2. **モデルの役割分担を崩さない**:
   - 記事1本の要約 = `MODEL_CHEAP`(Haiku)。1記事は独立した小タスクなので安いモデルで十分。
   - 最終ダイジェスト編集 = `MODEL_EDITOR`(Sonnet)。取捨選択と構成が必要な工程のみ。
   - Opus等の上位モデルを実行時に使うコードを書かない。品質が足りない場合はモデルではなく**プロンプト(systemの出力形式指定)を直す**。
3. **部分失敗で全体を止めない**: フィード1本の失敗、記事1本の要約失敗は、ログを出してフォールバックし継続する。全滅時のみ非ゼロ終了。
4. **state.json は送信成功後にのみ保存**: 送信前に保存すると、送信失敗時にその記事が二度と通知されなくなる。この順序を変えない。
5. **秘密情報は環境変数のみ**: `ANTHROPIC_API_KEY`, `SLACK_WEBHOOK_URL` をコード・config・ログに書かない。

## 2.5 フィード戦略(媒体を追加・変更するときの判断基準)

収集元は「公式RSSがあれば公式、なければGoogle News RSS経由」の2段構え。現在の構成:

| 媒体 | 方式 | 理由 |
|---|---|---|
| CNN / BBC | 公式RSS | 公式フィードが健在 |
| ロイター | Google News経由 | 2020年に公式RSSを廃止 |
| Bloomberg | Google News経由 | 公式の公開RSSなし |
| 時事通信 | Google News経由 | 公式RSSはあるが規約が「個人の私的利用のみ・プログラムによる再配信不可」のため直接取得を避ける |
| 福島民友 | Google News経由 | 公式RSSが確認できない |

Google News RSSのURL形式(これをテンプレとして使う):
```
https://news.google.com/rss/search?q=when:24h+allinurl:<ドメイン>&hl=ja&gl=JP&ceid=JP:ja
```
- `when:24h` = 直近24時間、`allinurl:` = そのドメインの記事に限定
- Google News経由の記事タイトルは末尾に「 - 媒体名」が付くが、`collect.py` の `_clean_google_news_title` が自動除去する(news.google.comのリンクのみ対象。通常フィードのタイトルは触らない)
- 媒体を追加するときは、まず公式RSSの有無と**利用規約**を確認してから方式を決める。規約に「再配信禁止」「私的利用限定」とあればGoogle News経由にする

## 3. 変更手順(この順番を機械的に実行する)

どんな変更でも、以下を上から順に実行する。スキップ禁止。

1. `python3 -m unittest discover -s tests` を実行し、**変更前がGREENであること**を確認する。REDなら先にそれを直す。
2. 変更を実装する。1コミット1目的。
3. 変更に対応するテストを `tests/test_all.py` に追加する。**ネットワークを実際に叩くテストは書かない**(`urllib.request.urlopen` を必ずモックする)。
4. `python3 -m unittest discover -s tests` がGREENになるまで直す。
5. `python3 -m news_agent.main --dry-run` を実行し、クラッシュせず終了することを確認する(exit 0 または、フィード到達不能環境なら exit 2 が正しい)。
6. CHECKLIST.md の該当項目を確認する。

## 4. デバッグ規約(「動かない」と言われたときの決定木)

症状から順に辿る。**推測で複数箇所を同時に変更しない。1仮説→1修正→1テスト。**

| 症状 | 最初に確認 | 直し方 |
|---|---|---|
| 何も通知が来ない | `python -m news_agent.main --dry-run` の出力 | ログの `[collect]` 行でどのフィードが失敗しているか特定 |
| `新着なし` と出る | `state.json` | テスト中なら `rm state.json` で既読リセット |
| `ANTHROPIC_API_KEY が未設定` | 環境変数 / GitHub Secrets | キーを設定。コードは触らない |
| APIエラー 401 | キーの値 | キー再発行。コードは触らない |
| APIエラー 429/529 が頻発 | `main.py` の `time.sleep(0.3)` | 待機を増やす(0.5〜1.0)。リトライ回数は増やさない(コスト増) |
| Slack送信失敗 | Webhook URLの有効性(`curl`で直接POST) | URL再発行。`notify.py` は触らない |
| 特定フィードだけ0件 | そのフィードのXMLを `curl` で取得し目視 | RSS/Atom以外の形式なら `parse_feed` に分岐追加+テスト追加 |
| ダイジェストの体裁が崩れる | `summarize.py` の `build_digest` のsystemプロンプト | 出力形式の指定を具体化する。コード側で整形しようとしない |
| Google News経由の媒体だけ0件 | ブラウザでそのフィードURLを開く | `when:24h` を `when:48h` に緩める/`allinurl:` のドメイン表記を確認(例: bloomberg.co.jp と bloomberg.com の違い) |
| タイトルに「 - 媒体名」が残る/消えすぎる | `collect._clean_google_news_title` | news.google.comリンクのみ対象という条件を維持したままテストを追加して直す |

## 5. 要約品質の維持(モデルが弱くても品質を保つ仕組み)

品質はモデルの賢さではなく**プロンプトの出力形式指定**で担保する。以下を守る:

- systemプロンプトには必ず「出力形式(厳守)」節を置き、行単位でフォーマットを指定する。
- 「前置き不要」「要約文だけを出力」を明記する(小型モデルは前置きを付けがち)。
- 文字数・文数の上限を数値で指定する(「簡潔に」は禁止。「最大2文」と書く)。
- 出力が崩れたら、悪い出力例を見て「〜しない」という否定形の禁止事項をsystemに1行追加する。

## 6. コスト規約

- 1記事の要約入力は本文1500文字まで(`summarize_article` 内で切り詰め済み。上げない)。
- `max_articles` はデフォルト10。増やす場合はHaiku呼び出し回数=記事数であることを意識する。
- ダイジェスト生成(Sonnet)は1日1回のみ。ループ内でSonnetを呼ぶコードを書かない。

## 7. やってはいけないこと(過去の失敗の再発防止)

- デフォルト引数に `path=STATE_PATH` のようなモジュール定数を直接書かない(定義時束縛でテスト・差し替えが効かない)。`None` にして関数内で解決する。
- テストで実URLを叩かない。CIで不安定化する。
- 例外を握りつぶして `pass` しない。必ずログを出す。
- Slackのsectionブロックに3000文字超を入れない(`_chunk_text` を経由する)。
- テストのbytesリテラル(`b"""..."""`)に日本語を直接書かない(SyntaxErrorになる)。文字列で書いて `.encode("utf-8")` する。
- フィードのパースをタグの完全一致(`root.iter("item")`)で書かない。RSS 1.0は名前空間付きなのでマッチしない。`_local()` によるローカル名判定を使う。
